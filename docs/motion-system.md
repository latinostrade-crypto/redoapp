# Motion-система интерактивного комикса

## 1. Исходные условия и решение по стеку

Проект уже работает на React 19, Vite 6, TypeScript и Tailwind CSS 4. Библиотека
`motion` используется внутри существующей игры для локальных UI-переходов. В
проекте нет GSAP и ScrollTrigger.

Для скролл-режиссуры следует добавить `gsap` и использовать ScrollTrigger из
этого же пакета. Это оправданная новая зависимость: требуются обратимые
scrub-timeline, pinning, точный контроль прогресса, адаптивные сценарии и
контролируемый `refresh`. `motion` сохраняется для уже существующих игровых
модалок и небольших независимых UI-анимаций, но не управляет элементами
scroll-сцен. Один DOM-узел не должен одновременно получать `transform` или
`opacity` от GSAP и Motion.

GSAP следует загружать динамически только при открытии комикса. Игровой экран
не должен оплачивать размер ScrollTrigger и не должен монтироваться за
комиксом. Обратное тоже верно: при входе в игру все scene timeline и trigger
уже уничтожены.

Критические ограничения текущего приложения:

- `src/App.tsx` — монолитный игровой экран со своим состоянием, таймерами,
  загрузчиком, wallet/realtime-потоками и `overflow`-режимами;
- `src/main.tsx` сейчас сразу вызывает `wakeBackend()` и предзагружает карты;
- Telegram `startapp=room_*` и `startapp=ref_*` должны по-прежнему сразу
  попадать в рабочий игровой flow;
- React работает в `StrictMode`, поэтому setup и cleanup анимаций должны быть
  симметричными и выдерживать двойной mount в development;
- глобальное `image-rendering: pixelated` уже соответствует стилю ассетов.

Рекомендуемая граница: небольшой `AppShell` выбирает комикс или игру, а сам
комикс и существующий `GameApp` загружаются отдельными чанками. Любой валидный
Telegram launch parameter обходит комикс. Все игровые CTA комикса открывают
реальный Telegram Mini App по адресу
`https://t.me/redo_appbot/app?startapp=ref_KNVPOU`; они не являются
декоративными кнопками. Не следует держать игру невидимо смонтированной.

## 2. Что позволяют реальные изображения

В `FOR AI/WEBSITE` находятся 12 непрозрачных JPEG, все размером `768 × 1376`
(`aspect-ratio: 768 / 1376`). Это цельные кадры без alpha-канала, поэтому
«разбор на слои» нельзя имитировать вырезанными персонажами. Честные приёмы:
кадрирование, дубли одного кадра в разных `overflow: hidden`-окнах, маски,
изменение `object-position`, перекрытие соседним цельным кадром и SVG/CSS-декор.

Визуальные пары дают естественные match cut:

- три окна под дождём: девушка, барсук и Pepe — почти одинаковая рама;
- два коридора с колоннами: девушка и идущий барсук;
- общий зал с девушкой и барсуком продолжает коридор;
- девушка в лесу и девушка на троне имеют сильный центральный силуэт;
- циклоп на каменном полу и циклоп на пирсе имеют центральную посадку;
- Pepe с сердцем за окном и под красной луной связываются формой сердца;
- барсук на балконе даёт сильный вертикальный финальный или конфликтный кадр.

Из этого набора рекомендуется строить 6–7 глав разной длины. Motion-конфигурация
не должна жёстко зависеть от названий глав: storyboard задаёт порядок, а
motion preset — только способ движения.

## 3. Архитектура

Предлагаемая структура ответственности:

```text
ComicExperience
├── StoryProgress
├── ComicScene × N
│   ├── SceneStage
│   ├── ImageFrame / ImageTransition
│   ├── ComicPanel
│   ├── Caption / SpeechBubble / SoundEffect
│   └── SpeedLines / ImpactBurst
└── FinalCTA
```

Конфигурация сцены хранит только сериализуемые данные:

- `id`, `chapter`, `scrollVhDesktop`, `scrollVhMobile`;
- список кадров и их `src`, `alt`, `objectPosition`;
- `motionPreset` и `transitionPreset`;
- нормализованные ключевые точки от `0` до `1`;
- тексты, визуальный тип реплики и адаптивные позиции;
- отдельные `desktop` и `mobile` параметры;
- статическое состояние для reduced motion.

Компоненты не решают сюжетных вопросов и не вызывают React `setState` на
каждом `onUpdate`. Текущий номер главы можно обновлять только при пересечении
границы сцены; вся покадровая работа остаётся внутри GSAP.

`SceneStage` содержит:

1. внешний scroll-контейнер, задающий длину главы;
2. один закрепляемый viewport;
3. внутренний `camera`, на котором выполняются dolly/pan;
4. самостоятельные панели и типографику поверх камеры.

У закрепляемого элемента не должно быть transform-предка. Нельзя включать
глобальный smooth-scroll, заменять native scroll или ставить `overflow:
hidden` на `body` на всём протяжении истории.

## 4. Жизненный цикл React и ScrollTrigger

Каждая сцена создаёт ровно одну master timeline внутри `useLayoutEffect`.
Timeline не проигрывается самостоятельно и получает один ScrollTrigger:

```ts
const mm = gsap.matchMedia();
const ctx = gsap.context(() => {
  mm.add("(prefers-reduced-motion: no-preference)", () => {
    const timeline = gsap.timeline({
      defaults: { ease: "none" },
      scrollTrigger: {
        trigger: scene,
        start: "top top",
        end: "bottom top",
        pin: stage,
        pinSpacing: false,
        scrub: isMobile ? 0.25 : 0.55,
        anticipatePin: 1,
        invalidateOnRefresh: true,
      },
    });
    buildSceneTimeline(timeline, refs, config);
  });
}, rootRef);

return () => {
  mm.revert();
  ctx.revert();
};
```

Это схема, а не код для буквального копирования. В реальной реализации важно:

- сохранять `mm`, `ctx` и принадлежащие сцене trigger в замыкании;
- не вызывать глобальный `ScrollTrigger.killAll()`: он может уничтожить
  чужие trigger;
- cleanup обязан снять listeners, `ResizeObserver`, preload callbacks и
  временный `will-change`;
- после `ctx.revert()` DOM должен вернуться в исходное CSS-состояние;
- регистрация плагина выполняется один раз после client-only import;
- идентификаторы trigger должны включать `scene.id`, чтобы диагностика была
  читаемой.

## 5. Правила полностью обратимых timeline

Все визуальные состояния зависят только от прогресса master timeline.
Запрещено собирать сцену через необратимые `onEnter`/`onLeave` с мутацией
React-state.

- Использовать явные `fromTo`, а не полагаться на текущее вычисленное
  состояние.
- Не использовать `Math.random()` внутри timeline. Координаты линий и shake
  задаются конфигурацией или seed-данными.
- Тексты присутствуют в DOM до запуска; timeline меняет только их визуальное
  состояние.
- Для смены `display` предпочтительнее `autoAlpha`; интерактивность CTA
  включается только в финальном состоянии через небольшой, обратимый
  дискретный callback либо всегда остаётся доступной в нормальном потоке.
- На границе сцен кадр A должен иметь точно заданное конечное состояние, а
  кадр B — точно заданное начальное. Нельзя зависеть от того, с какой скоростью
  пользователь вошёл в сцену.
- Для быстрого скролла timeline должна корректно перескочить в конечное
  состояние; не запускать отдельные не-scrubbed анимации из `onEnter`.

Рекомендуемый ритм одной главы:

| Прогресс | Событие |
| --- | --- |
| `0.00–0.12` | принятие предыдущего кадра и фиксация композиции |
| `0.12–0.42` | движение камеры или раскрытие панели |
| `0.34–0.62` | вход реплики и вторичного визуального акцента |
| `0.58–0.82` | sound effect или сюжетный поворот |
| `0.78–1.00` | image-to-image transition и подготовка следующей сцены |

Диапазоны намеренно пересекаются. Реальные главы должны отличаться по ритму.

## 6. Каталог motion preset

### Rain-window match cut

Для трёх кадров за одинаковой рамой. Рама остаётся визуальным якорем, а новый
кадр раскрывается вертикальными полосами, повторяющими потёки воды. На desktop
допустимо 6–8 полос; на mobile — одна `clip-path`-шторка или translate-reveal.
Это не обычный crossfade: старый кадр виден между полосами до конца wipe.

### Hallway split

Для двух коридоров. Камера медленно приближается к тёмной центральной точке,
затем диагональная граница или центральный split заменяет девушку на барсука.
Колонны остаются совпадающими по масштабу. На завершении кадр отъезжает в общий
зал с двумя персонажами. Масштаб не превышает примерно `1.12`, чтобы не
раздувать текстуру.

### Center iris

Для леса/трона и пары циклопа. Следующий кадр раскрывается круговой маской из
центра композиции. В Safari использовать совместимый `-webkit-mask` fallback
или вложенный контейнер с `clip-path: circle()`. Mobile fallback — масштаб
круглой панели без анимации сложной mask-position.

### Heart occlusion

Переход между двумя кадрами Pepe строится вокруг уже нарисованного сердца.
Нельзя вырезать сердце как будто оно отдельный PNG. Вместо этого второй цельный
кадр появляется внутри заранее заданной CSS/SVG heart-mask, после чего маска
расширяется и заполняет сцену. В simplified mobile режиме используется
обычный центральный polygon/circle reveal.

### Comic page tear

Чёрная CSS/SVG-панель с рваной границей проходит по экрану и закрывает смену
кадра. Достаточно 12–16 заранее заданных точек polygon; генерировать сотни
кусочков не нужно. Это подходящий переход к балкону или из него.

### Fine particle dust

Каждая normal-motion смена использует один переиспользуемый canvas-слой на всю
сцену. Поле состоит примерно из `14000` image + `4500` interface частиц на mobile
и `22000` image + `6500` interface частиц на desktop. Размер движущейся песчинки держится
около `0.45–1.75 px`. Цвета image-поля семплируются из реального цельного кадра;
interface-поле растеризует геометрию HTML-блоков, текста, bubbles, facts, SFX и
CTA. Главный scene copy помечен как `primary`: он поднимается и собирается в
той же фазе, что и изображение. Comic inserts не входят во входящее particle
field: после сборки кадра bubbles, facts, SFX и CTA появляются по одному через
собственные pop, slide и impact tweens. Поэтому исчезает не декоративный слой поверх картинки, а вся видимая
композиция, после чего следующий кадр и следующий набор уже раскрытых текстов
собираются из собственной цветной пыли.

Позиции и траектории зависят от seed сцены, поэтому один и тот же прогресс всегда
даёт один и тот же кадр и корректно проигрывается назад. Переход использует не
два независимых облака, а единый цикл материала. В момент `0` исходные DOM-слои
скрываются и без визуального разрыва заменяются полностью замощённым outgoing
particle field. Затем `0–0.08` — сжатие его плиток до мелких песчинок и раскол,
`0.08–0.42` — падение всех песчинок ниже
stage, `0.42–0.72` — подъём и сборка нового изображения, `0.66–0.94` — более
поздний подъём HTML-текста и блоков. Реальный incoming image включается только
на `0.78`, после полной particle-сборки; HTML передаётся DOM на `0.96`.
Между главами ownership разделён: уходящая сцена семплирует и рассыпает только
собственный последний кадр, а входящая сцена сама собирает из нижнего облака
свой первый кадр и primary copy. Canvas соседней главы никогда не рисует копию
чужого DOM, поэтому на стыке не может появиться сломанный дубликат.

Canvas рисуется только внутри активного окна перехода и не создаёт particle
DOM-узлы. На слабых устройствах плотность снижается до `62%`, дополнительные
хвосты отключаются, но песчинки не заменяются крупными блоками. В зависимости
от количества кадров internal cycle занимает `16–26%` timeline; межсценное
падение занимает последние `12%`, а owned entrance — первые `9%` desktop или
`10.5%` mobile следующей главы. Пыль обеих фаз встречается у нижнего края stage,
поэтому между ними нет чёрного hold. Физическая scroll-длина каждой главы равна
`275–320vh` desktop / `280–315svh` mobile. Pin завершается ровно в момент,
когда следующая глава достигает `top`, поэтому между главами нет лишнего
viewport с двумя одновременно видимыми stages. Primary copy не участвует во
внутренних сменах изображений и остаётся читаемым; из частиц он собирается один
раз — во время owned entrance своей главы. Начало первой сцены сразу показывает
уже собранный первый кадр и copy. Scroll scrub равен
`0.62` mobile / `0.9` desktop. В reduced-motion режиме particle dust полностью
отсутствует.

### Comic-panel slide

Два или три кадрированных экземпляра цельного изображения входят с разных
сторон в рамках с чёрной обводкой. Движется контейнер панели, не само
полноэкранное изображение. Панели имеют разные углы не более `2–3deg`, чтобы
пиксельная графика оставалась чёткой.

## 7. Камера, параллакс и типографика

Глубина создаётся без ложного вырезания объектов:

- background: цельный кадр, `scale 1.03 → 1.10`;
- midground: кадрированный дубликат в узкой панели, `yPercent ±2–4`;
- foreground: CSS/SVG-рамка, чёрная шторка или крупная панель,
  `yPercent ±5–8`;
- typography: независимый слой с минимальным движением после появления.

Одновременно активны не более 2–4 уровней. На mobile остаются 1–2. Камера
движется плавным `sine.inOut`/`power1.inOut`, объектный вход — `power3.out`,
реплика — коротким overshoot (`back.out`), удар — жёсткими keyframes.
Master scrub не обязан иметь общий easing `none`: `none` остаётся default, а
внутренним tween задаются характерные easing.

Speech bubble и caption — семантический DOM-текст. SVG/CSS отвечает только за
форму. Реплика появляется через `scale: 0.72 → 1`, `rotation: ±2deg → 0` и
короткое затухание; на mobile без word-by-word stagger.

Sound effect:

- существующие Press Start 2P/Silkscreen подходят по стилю;
- обводка и offset-shadow создаются CSS, а не дорогим filter;
- burst и speed lines имеют `aria-hidden="true"`;
- надпись появляется за `6–10%` прогресса главы и не мигает;
- допускаются `WHOOSH!`, `OOPS!`, `REDO!`, `READY?`, `GO!`, если они
  соответствуют утверждённому storyboard.

## 8. Impact moment

Один сильный impact лучше нескольких одинаковых. Рекомендуемая обратимая
последовательность:

1. `0.000–0.035` — чёрная шторка до `opacity: 0.75`;
2. `0.035–0.055` — один светлый flash до `opacity: 0.8` и обратно;
3. `0.045–0.095` — snap zoom камеры примерно `1.00 → 1.08`;
4. `0.055–0.115` — заранее заданный shake `x: [0,-6,5,-3,0]`,
   `y: [0,2,-2,1,0]`;
5. `0.075–0.18` — крупное слово с overshoot;
6. `0.09–0.22` — 8–12 SVG speed lines;
7. `0.18–0.30` — камера возвращается к `1.02`;
8. далее начинается следующий transition.

Из-за scrub flash никогда не должен повторяться автоматически. При
`prefers-reduced-motion` и в low-power mobile preset остаются статическое слово
и чёрный comic-panel cut, без flash, zoom и shake.

## 9. Desktop, mobile и Telegram presets

### Desktop

- главы: примерно `260–460svh`, разные по длине;
- `scrub: 0.45–0.7`;
- до трёх панелей одновременно;
- параллакс до `8%`, угол камеры до `1deg`;
- сложная mask-анимация разрешена только одной сцене одновременно.

### Mobile / coarse pointer

- главы: примерно `180–300svh`;
- `scrub: 0.2–0.35`, чтобы интерфейс не «догонял» палец;
- одна основная панель и максимум одна вторичная;
- параллакс до `3–4%`, без постоянного rotation;
- clip/mask заменяется transform-шторкой, если устройство не держит кадр;
- нативный `pan-y`, без зависимости от hover.

### Telegram WebView

- не запрашивать fullscreen для комикса автоматически; текущего `expand()`
  достаточно;
- закреплённая сцена использует стабильную высоту Telegram viewport, а
  интерактивные элементы дополнительно учитывают `contentSafeAreaInset`;
- слушать `viewportChanged`; дорогостоящий refresh делать только после
  стабильного состояния;
- на событии `activated` и возврате из wallet обновить viewport и один раз
  вызвать контролируемый refresh;
- не менять Back Button или launch routing ради декоративной навигации.

## 10. Viewport, resize и refresh

Геометрию pin следует считать от стабильной высоты, а не от каждого изменения
адресной строки. Источник высоты по приоритету:

1. Telegram `viewportStableHeight`, если доступен;
2. CSS `100svh` для мобильной pin-геометрии;
3. `visualViewport.height` после стабилизации;
4. `100vh` как fallback.

`100dvh` полезен для визуального заполнения текущего окна, но не должен менять
scroll distance на каждый пиксель движения панели Safari.

Контролируемый refresh выполняется:

- после `document.fonts.ready`;
- после `decode()` изображений текущей и следующей сцены;
- после устойчивого `orientationchange`;
- после Telegram `viewportChanged` с `isStateStable`;
- после реального изменения ширины или stable-height, а не каждого события
  mobile browser chrome.

События объединяются одним `requestAnimationFrame` и debounce порядка
`120–200ms`. Перед refresh выполняются все чтения размеров, затем все записи.
Не использовать ручной scroll listener для измерения layout.

## 11. Reduced motion и доступность

`gsap.matchMedia()` разделяет normal и reduced-motion ветки. Reduced motion —
не timeline с нулевой длительностью, а нормальная вертикальная статья:

- нет pin и scrub;
- все изображения и тексты находятся в документном потоке;
- один кадр на секцию, возможен короткий opacity transition;
- нет parallax, shake, flash, particle dissolve и whip-pan;
- сюжет, заголовки и CTA полностью сохранены;
- progress становится обычным списком/номером главы.

Доступность normal режима:

- у каждой главы есть настоящий heading и описательный текст;
- важный смысл не хранится только внутри JPEG;
- декоративные дубли изображений имеют пустой `alt` и `aria-hidden`;
- активный CTA имеет видимый `:focus-visible` и touch-target не менее 44 px;
- scroll progress не объявляется через `aria-live` при каждом пикселе;
- порядок Tab соответствует DOM, а не визуальным абсолютным позициям;
- отсутствуют autoplay-звук, частый flash и перехват клавиатуры.

## 12. Чек-лист реализации

- [ ] GSAP/ScrollTrigger находится только в lazy comic chunk.
- [ ] Игра и комикс не смонтированы одновременно.
- [ ] Telegram `startapp` сразу открывает существующий flow.
- [ ] У каждой сцены один master timeline и один владелец cleanup.
- [ ] Скролл вверх полностью восстанавливает все состояния.
- [ ] Нет global `killAll`, глобального `body.style.overflow` и custom scroll.
- [ ] Все 12 JPEG используются как цельные честные кадры или кадрированные
      дубли; несуществующие alpha-слои не выдуманы.
- [ ] Реализовано минимум четыре разных transition preset.
- [ ] Impact moment только один, детерминированный и безопасный.
- [ ] Mobile preset короче и проще desktop, а не просто масштабирован.
- [ ] Reduced-motion версия читается без GSAP.
- [ ] Refresh не запускается непрерывно при движении browser chrome.
